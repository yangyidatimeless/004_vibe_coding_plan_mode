## 🧪 QA 视角的建议

### 1. 测试覆盖计划

Plan 模式的核心是"可预测性"，这正好与测试理念契合。我建议：

**单元测试层面**：
- 每个步骤的执行逻辑需要独立的单元测试
- 异常处理分支的测试覆盖率要达到 100%
- Mock 外部依赖（文件系统、API 调用等）确保测试隔离

**集成测试层面**：
- 完整流程的 E2E 测试（从计划生成→用户确认→执行→完成）
- 异常场景测试（文件冲突、权限不足、网络中断等）
- 回滚机制测试（某步骤失败后能否正确回退）

**UI 测试层面**：
- 状态图标的正确性验证（✓/○/⚠ 与实际状态一致）
- 实时更新测试（执行过程中 UI 能否正确反映进度）
- 可访问性测试（screen reader、键盘导航）

### 2. 质量门禁

建议在 Plan 模式中加入质量检查点：

```
┌─────────────────────────────────────────────────┐
│  🔍 质量检查                                    │
├─────────────────────────────────────────────────┤
│  □ 代码规范检查 (lint) 通过                    │
│  □ 单元测试通过率 > 80%                        │
│  □ 无高危安全漏洞                              │
│  □ 文档已更新                                  │
├─────────────────────────────────────────────────┤
│  [跳过检查] [执行检查] [查看详情]              │
└─────────────────────────────────────────────────┘
```

**执行前检查**：
- 用户可以选择是否在计划执行前自动运行测试
- 如果测试失败，给出警告但允许用户选择是否继续

**执行后验证**：
- 每一步执行完成后，自动验证预期结果是否达成
- 如果验证失败，立即暂停并提示用户

### 3. 错误报告与日志

从 QA 角度，错误信息需要足够详细以便定位问题：

**错误报告应包含**：
- 具体是哪一步骤失败
- 失败的详细原因（错误堆栈、输入参数、环境信息）
- 建议的解决方案（如果有）
- 一键导出错误日志（方便提交 issue）

**日志分级**：
- `INFO`: 正常执行流程
- `WARN`: 可恢复的异常（如文件已存在但内容相同）
- `ERROR`: 导致步骤失败的错误
- `DEBUG`: 详细的调试信息（用户可选择开启）

### 4. 回归测试策略

Plan 模式迭代过程中需要注意：
- 每次 UI 交互变更后，需要回归测试核心流程
- 建立自动化测试 CI，每次提交自动运行
- 保留历史测试报告，方便追踪质量问题

### 1. 回应美娜的问题：Python vs Go

**我的建议：选择 Python**，理由如下：

**团队技术栈匹配**：
- 少平舅舅前端用 Node.js，Python + Node 的组合在 Web 开发中非常成熟
- 美娜妈妈统筹管理，Python 生态丰富，快速迭代能力强
- 少锋舅舅 QA 测试，Python 的 pytest/unittest 生态完善，测试工具链成熟

**AI 集成优势**：
- AI 相关库（LangChain、LlamaIndex、transformers）都是 Python 优先
- Plan 生成需要调用大模型 API，Python 的 httpx/aiohttp 生态成熟
- 后续如果要本地部署小模型，Python 是唯一选择

**开发效率**：
- Python 代码简洁，快速验证想法
- 类型提示（Type Hints）+ pyright 可以保证代码质量
- FastAPI 框架开发 API 效率高，自动生成 OpenAPI 文档

**Go 的优势场景**（供参考）：
- 高并发、高性能场景（但我们初期用户量不大）
- 系统级编程、CLI 工具
- 如果需要独立部署的二进制文件

**结论**：Phase 1-3 都用 Python，如果后续有性能瓶颈再考虑用 Go 重写热点模块。

### 2. 架构设计思路
